Introducción

A continuación se describirá de forma completa toda la solución en fase inicial (primera itereración) para el problema propuesto como Assessment en Pragma. El problema en cuestión es el Sistema de Gestión de Torneosde Videojuegos. Todo esto desde la concepción inicial, gestión del proyecto, repositorios, tecnologías, etc.

Definición Funcional

En esta sección se explica toda la definición funcional, incluyendo las entidades del dominio y los casos de uso, tanto los que están actualmente implementados como los que no.

Descripción de las entidades

Donation

Está relacionada a todas las donaciones que se hagan para un torneo específico.

Game

Esta entidad contiene la información de todos los videojuegos que se pueden jugar en un torneo particular.

🟣 Value Objects
  • Classification: Enum - PEGI3 PEGI7 PEGI12 PEGI16 PEGI18
  • Console: Enum - PLAYSTATION XBOX PC SWITCH

Match

Está relacionada a las partidas que se van a jugar en un torneo en particular. Relaciona a los participantes que se van a enfrentar y contiene también al ganador de dicha partida cuando ésta finalice.

🟣 Value Objects
  • Status: Enum - NOT_STARTED IN_PROGRESS FINISHED CANCELED

Registration

Contiene la información relacionada al registro de un participante en un torneo. Sirve para tener claro si un participante se registró o no a un torneo particular.

🟣 Value Objects
  • Status: Enum - PENDING APPROVED CANCELED ENDED

Reward

Contiene información sobre el podio de participantes y sus respectivos premios monetarios

Ticket

Esta entidad contiene toda la información relacionada a los ticket de participación y visualización que se pueden adquirir.

🟣 Value Objects
  • Status: Enum - NEW USED BLOCKED
  • Type: Enum - VISUALIZATION PARTICIPATION

Tournament

Esta es la entidad principal del sistema. Todo gira en torno a los torneos. Contiene información de participantes, moderadores, y todo lo relacionado con la realización de un torneo.

🟣 Value Objects
  • Status: Enum - NOT_STARTED IN_PROGRESS FINISHED CANCELED
  • Commission: Object
    • VisualizationPercentage: Float
    • ParticipationPercentage: Float
    • DonationPercentage: Float
  • Price: Object
    • Participation: BigDecimal
    • Visualization: BigDecimal
  • Transmission: Object
    • Platform: Enum - TWITCH YOUTUBE DISCORD
    • url: String

Transaction

A tráves de esta entidad se puede llevar el registro de las transacciones y pagos sobre los tickets que se adquieran o las donaciones que se realicen.

🟣 Value Objects
  • Status: Enum - APPROVED REJECTED IN_PROCESS STARTED
  • Type: Enum - VISUALIZATION PARTICIPATION DONATION

User

Son las entidades que harán uso del sistema. Pueden visualizar torneos, participar en ellos, o gestionarlos (dependiendo de su Rrol).

🟣 Value Objects
  • Status: Enum - ACTIVE BANNED
  • Role: Enum - VIEWER PROMOTER ADMINISTRATOR PARTICIPANT MODERATOR

Casos de uso

A continuación se listarán todos los casos de uso que se plantearon para la solución junto con las reglas de negocio asociadas

Tournament

  • Registrar un torneo
    • El usuario que registra un torneo no puede tener estado BANNED
    • El usuario que registra un torneo debe tener rol ADMINISTRATOR o PROMOTER
    • Un usuario ADMINISTRATOR puede generar la cantidad de torneos gratis o de pago que desee
    • Si el usuario es PROMOTER y el torneo es gratis, se validará si aún no supera la cantidad máxima de torneos gratis permitidos
    • Registrar el torneo con estado NOT_STARTED
    • La cantiadad de plazas disponibles será igual al campo de plazas máximas
    • Notificar al promotor via SMS y/o email el registro exitoso del torneo
  • Empezar un torneo
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol ADMINISTRATOR, PROMOTER o MODERATOR. Para estos dos últimos, el rol lo deben tener dentro del torneo en cuestión
    • Validar que el estado del torneo no sea FINISHED, CANCELED o IN_PROGRESS
    • Validar que el torneo ya haya alcanzado la cantidad de plazas mínimas de participación
    • Validar que la fecha de inicio del torneo efectivamente ya se alcanzó
    • Actualizar el estado del torneo a IN_PROGRESS
    • Notificar al promotor via SMS y/o email la puesta en marcha del torneo
    • Notificar a los moderadores via SMS y/o email la puesta en marcha del torneo
    • Notificar a los participantes via SMS y/o email la puesta en marcha del torneo
    • Notificar a los espectadores via SMS y/o email la puesta en marcha del torneo
  • Cancelar un torneo
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol ADMINISTRATOR o PROMOTER. Para este último, el rol lo debe tener dentro del torneo en cuestión
    • Validar que el estado del torneo no sea FINISHED o CANCELED
    • Actualizar el estado del torneo a CANCELED
    • Notificar al promotor via SMS y/o email la cancelación del torneo
    • Notificar a los moderadores via SMS y/o email la cancelación del torneo
    • Notificar a los participantes via SMS y/o email la cancelación del torneo
    • Notificar a los espectadores via SMS y/o email la cancelación del torneo
  • Finalizar un torneo
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol ADMINISTRATOR, PROMOTER o MODERATOR. Para estos dos últimos, el rol lo deben tener dentro del torneo en cuestión
    • Validar que el estado del torneo no sea NOT_STARTED, FINISHED o CANCELED
    • Validar que la fecha de fin del torneo efectivamente ya se alcanzó
    • Validar que no existan Matches asociadas a este torneo que estén en estado NOT_STARTED o IN_PROGRESS
    • Actualizar el estado del torneo a FINISHED
    • En la petición debe estar presente el premio asociado al torneo, en donde se relacione la tabla de posiciones
    • Notificar al promotor via SMS y/o email la finalización del torneo
    • Notificar a los moderadores via SMS y/o email la finalización del torneo
    • Notificar a los participantes via SMS y/o email la finalización del torneo
    • Notificar a los espectadores via SMS y/o email la finalización del torneo
  • Reagendar un torneo
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol ADMINISTRATOR, PROMOTER o MODERATOR. Para estos dos últimos, el rol lo deben tener dentro del torneo en cuestión
    • Actualizar las fechas de inicio y/o fin
    • Notificar al promotor via SMS y/o email el reagendamiento del torneo
    • Notificar a los moderadores via SMS y/o email el reagendamiento del torneo
    • Notificar a los participantes via SMS y/o email el reagendamiento del torneo
    • Notificar a los espectadores via SMS y/o email el reagendamiento del torneo
  • Agregar un moderador a un torneo
    • El usuario realiza la acción no puede tener estado BANNED
    • El o los usuarios que se agregarán como MODERATOR no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol ADMINISTRATOR, PROMOTER o MODERATOR. Para estos dos últimos, el rol lo deben tener dentro del torneo en cuestión
    • Validar que la cantidad de moderadores a agregar más los que ya tiene el torneo no superen la cantidad máxima establecida
    • Actualizar el torneo con los nuevos moderadores
    • Notificar al promotor via SMS y/o email la inclusión de un nuevo moderador
    • Notificar a los moderadores via SMS y/o email la inclusión de un nuevo moderador
  • Eliminar un moderador de un torneo
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol ADMINISTRATOR o PROMOTER. Para este últimos, el rol lo debe tener dentro del torneo en cuestión
    • Actualizar el torneo eliminando al moderador
    • Notificar al promotor via SMS y/o email la eliminación del moderador
    • Notificar a los moderadores via SMS y/o email la eliminación del moderador
  • Pre-registrar un participante
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol PARTICIPANT
    • Validar que el usuario no tenga un pre-registro en este torneo o que ya esté como participante
    • Validar que el usuario no esté en la lista de descalificados
    • Validar que el torneo no haya iniciado aún, que esté cancelado o que ya haya terminado. Que no esté en estado IN_PROGRESS, FINISHED o CANCELED
    • Si el torneo es de pago, crear un registro de Transaction de tipo PARTICIPATION y estado STARTED
    • Si el torneo es de gratis, crear un registro de Transaction de tipo PARTICIPATION y estado APPROVED
    • Crear un registro en Registration con estado PENDING
    • Actualizar, para el torneo, la lista de pre-registros del torneo agregando al nuevo pre-registrado
    • Actualizar en el torneo la cantidad de plazas restantes, restando este número en 1
    • Notificar al promotor via SMS y/o email el pre-registro al torneo
    • Notificar a los moderadores via SMS y/o email el pre-registro al torneo
    • Notificar al pre-registrado via SMS y/o email el pre-registro al torneo
  • Registrar un participante previamente pre-registrado
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol PARTICIPANT
    • Validar que el usuario tenga un pre-registro en este torneo
    • Validar que el usuario no sea ya un participante activo
    • Validar que el usuario no esté en la lista de descalificados
    • Validar que el torneo no haya iniciado aún, que esté cancelado o que ya haya terminado. Que no esté en estado IN_PROGRESS, FINISHED o CANCELED
    • Validar que tenga un ticket en estado NEW
    • Actualizar el registro en Registration con estado APPROVED
    • Actualizar, para el torneo, la lista de pre-registros del torneo eliminando al participante
    • Actualizar, para el torneo, la lista de registros del torneo agregando al nuevo participante
    • Actualizar el estado del ticket a USED
    • Notificar al promotor via SMS y/o email el registro exitoso en el torneo
    • Notificar a los moderadores via SMS y/o email el registro exitoso en el torneo
    • Notificar al participante via SMS y/o email el registro exitoso en el torneo
  • Pre-registrar un espectador
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol VIEWER
    • Validar que el torneo no haya iniciado aún, que esté cancelado o que ya haya terminado. Que no esté en estado IN_PROGRESS, FINISHED o CANCELED
    • Si el torneo es de pago, crear un registro de Transaction de tipo VISUALIZATION y estado STARTED
    • Si el torneo es de gratis, crear un registro de Transaction de tipo VISUALIZATION y estado APPROVED
    • Notificar al promotor via SMS y/o email el pre-registro de visualización torneo
    • Notificar a los moderadores via SMS y/o email el pre-registro de visualización torneo
    • Notificar al pre-registrado via SMS y/o email el pre-registro de visualización torneo
  • Registrar un espectador previamente pre-registrado
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol VIEWER
    • Validar que el torneo no haya iniciado aún, que esté cancelado o que ya haya terminado. Que no esté en estado IN_PROGRESS, FINISHED o CANCELED
    • Validar que tenga un ticket en estado NEW
    • Actualizar el estado del ticket a USED
    • Retornar la información de Transmissions perteneciente al torneo
    • Notificar al promotor via SMS y/o email el registro de visualización torneo
    • Notificar a los moderadores via SMS y/o email el registro de visualización torneo
    • Notificar al espectador via SMS y/o email el registro de visualización torneo
  • Descalificar un participante
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol ADMINISTRATOR, PROMOTER o MODERATOR. Para estos dos últimos, el rol lo deben tener dentro del torneo en cuestión
    • Validar que el torneo no esté en estado FINISHED o CANCELED
    • Validar que el participante esté en la lista de pre-registro o en la lista de participantes
    • Remover al usuario tanto de la lista de pre-registro como de la lista de participantes (cualquiera en la que esté)
    • Agregar al usuario a la lista de descalificados
    • Notificar al promotor via SMS y/o email la descalificación del torneo
    • Notificar a los moderadores via SMS y/o email la descalificación del torneo
    • Notificar al participante via SMS y/o email la descalificación del torneo

Match

  • Registrar una partida
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol ADMINISTRATOR, PROMOTER o MODERATOR. Para estos dos últimos, el rol lo deben tener dentro del torneo en cuestión
    • Validar que todos los usuarios que se van a registrar al torneo sean efectivamente participantes
    • Registrar la partida en estado NOT_STARTED
  • Cancelar una partida
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol ADMINISTRATOR, PROMOTER o MODERATOR. Para estos dos últimos, el rol lo deben tener dentro del torneo en cuestión
    • Validar que la partida no esté en estado FINISHED o CANCELED
    • Actualizar la partida a estado CANCELED
    • Notificar al promotor via SMS y/o email la cancelación de la partida
    • Notificar a los moderadores via SMS y/o email la cancelación de la partida
    • Notificar a los participantes via SMS y/o email la cancelación de la partida
    • Notificar a los espectadores via SMS y/o email la cancelación de la partida
  • Reagendar una partida
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol ADMINISTRATOR, PROMOTER o MODERATOR. Para estos dos últimos, el rol lo deben tener dentro del torneo en cuestión
    • Validar que la partida no esté en estado FINISHED o CANCELED
    • Actualizar la partida con la nueva fecha de inicio
    • Notificar al promotor via SMS y/o email el reagendamiento de la partida
    • Notificar a los moderadores via SMS y/o email el reagendamiento de la partida
    • Notificar a los participantes via SMS y/o email el reagendamiento de la partida
    • Notificar a los espectadores via SMS y/o email el reagendamiento de la partida
  • Empezar una partida
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol ADMINISTRATOR, PROMOTER o MODERATOR. Para estos dos últimos, el rol lo deben tener dentro del torneo en cuestión
    • Validar que la partida no esté en estado IN_PROGRESS FINISHED o CANCELED
    • Validar que la fecha de inicio ya haya llegado
    • Actualizar la partida con estado IN_PROGRESS y con la fecha y hora de inicio
    • Notificar al promotor via SMS y/o email el comienzo de la partida
    • Notificar a los moderadores via SMS y/o email el comienzo de la partida
    • Notificar a los participantes via SMS y/o email el comienzo de la partida
    • Notificar a los espectadores via SMS y/o email el comienzo de la partida
  • Terminar una partida
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol ADMINISTRATOR, PROMOTER o MODERATOR. Para estos dos últimos, el rol lo deben tener dentro del torneo en cuestión
    • Validar que el ganador esté registrado dentro de la partida
    • Validar que la partida no esté en estado NOT_STARTED FINISHED o CANCELED
    • Actualizar la partida con estado FINISHED, con la fecha y hora de finalización y con el id del ganador
    • Notificar al promotor via SMS y/o email la finalización de la partida
    • Notificar a los moderadores via SMS y/o email la finalización de la partida
    • Notificar a los participantes via SMS y/o email la finalización de la partida
    • Notificar a los espectadores via SMS y/o email la finalización de la partida

Ticket

  • Registrar un Ticket
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol VIEWER o PARTICIPANT
    • Validar que la transacción asociada al ticket tenga estado APPROVED (Significa que el pago ya fue hecho en caso de que el torneo fuera de pago)
    • Crear el QR asociado al ticket
    • Registrar un nuevo ticket en estado NEW y el QR previamente creado
    • Notificar al espectador o participante via SMS y/o email la creación del ticket y enviar la información de éste (junto con la imagen del QR) en el mismo correo
  • Bloquear un Ticket
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol ADMINISTRATOR, PROMOTER o MODERATOR. Para estos dos últimos, el rol lo deben tener dentro del torneo en cuestión
    • Validar que el ticket no haya sido bloqueado previamente
    • Notificar al espectador o participante via SMS y/o email la cancelación del ticket
  • Usar un Ticket
    • El usuario realiza la acción no puede tener estado BANNED
    • El usuario que realiza la acción debe tener rol VIEWER o PARTICIPANT
    • Validar que el ticket esté en estado NEW
    • Cambiar el estado del ticket a USED

Donation

  • Registrar una donación
    • El usuario realiza la acción no puede tener estado BANNED
    • Validar que el torneo no esté en estado FINISHED o CANCELED
    • Registrar un nuevo ticket en estado NEW y el QR previamente creado
    • Registrar la nueva donación junto con su transacción asociada

User

  • Registrar un nuevo usuario
    • Registrar el nuevo usuario con todos sus datos
  • Banear a un usuario
    • El usuario realiza la acción no puede tener estado BANNED
    • Quien realiza la acción debe ser un ADMINISTRATOR
    • Actualizar al usuario con el estado BANNED

Endpoints

Descargar - Colección Postman con todas las peticiones

Listado y descripción de los endpoints de la API. Para la variable {host} se deberá usar el host del API Gateway y no de los servicios aislados

Registrar un nuevo torneo: POST - {host}/api/v1/orchestrator/tournaments

Agregar uno o más moderadores: POST - {host}/api/v1/orchestrator/tournaments/{tournamentId}/moderators

Cancelar un torneo: PATCH - {host}/api/v1/orchestrator/tournaments/{tournamentId}/cancel

Descalificar a un participante: DELETE - {host}/api/v1/orchestrator/tournaments/{tournamentId}/users/{userId}

Finalizar un torneo: PATCH - {host}/api/v1/orchestrator/tournaments/{tournamentId}/finalize

Pre-registro: POST - {host}/api/v1/orchestrator/tournaments/{tournamentId}/users/{userId}/pre-register

Registrar participante: POST - {host}/api/v1/orchestrator/tournaments/{tournamentId}/users/{userId}/tickets/{ticketId}/register

Reagendar un torneo: PATCH - {host}/api/v1/orchestrator/tournaments/{tournamentId}/reschedule

Iniciar un torneo: PATCH - {host}/api/v1/orchestrator/tournaments/{tournamentId}/start

Registrar un ticket: POST - {host}/api/v1/orchestrator/tournaments/{tournamentId}/users/{userId}/tickets

Bloquear un ticket: PATCH - {host}/api/v1/orchestrator/users/{userId}/tickets/{ticketId}/block

Registrar una partida: POST - {host}/api/v1/orchestrator/tournaments/{tournamentId}/matches

Cancelar una partida: PATCH - {host}/api/v1/orchestrator/tournaments/{tournamentId}/matches/{matchId}/cancel

Finalizar una partida: PATCH - {host}/api/v1/orchestrator/tournaments/{tournamentId}/matches/{matchId}/finalize

Empezar una partida: PATCH - {host}/api/v1/orchestrator/tournaments/{tournamentId}/matches/{matchId}/start

Obtener un usuario: GET - {host}/v0/{tableId}/User/{id}

Obtener una lista de usuarios: GET - {host}/v0/{tableId}/User

Obtener una donacion: GET - {host}/v0/{tableId}/Donation/{id}

Obtener una lista de donaciones: GET - {host}/v0/{tableId}/Donation

Obtener un ticket: GET - {host}/v0/{tableId}/Ticket/{id}

Tecnologías Utilizadas

A continuación se listarán las tecnologías que se utilizaron para el desarrollo de la primera versión de la aplicación. Conforme se desarrollen las siguientes iteraciones, se estará actualizando el listado de tecnologías

API Gateway

El desarrollo del API Gateway se realizó utilizando el lenguaje de programación Java, junto con Spring Framework y, por último, para poder hacer el enrutado de forma correcta, se utilizó la dependencia de Spring Cloud Gateway.

La configuración de rutas se puede hacer de manera simple, usando Java, y con infinidad de condiciones de enrutado, tal como se muestra a continuación. Además, se pueden utilizar mecanismos de tolerancia a fallos tales como CircuitBreaker, Retry, FallBack, RateLimit, Bulkhead, entre otros.

En este caso hice la implementación de CircuitBreaker, Retry y FallBack

Descubrimiento de microservicios

Una parte importante en el manejo de microservicios es el autodescubrimiento de los mismos. Esto se hace posible a través de un servidor de descubrimiento de servicios. En este caso, se utilizó el que nos proporciona Netflix, llamado Eureka. Particularmente utilicé aquél que provee el paquete starter de Spring: spring-cloud-starter-netflix-eureka

Con este servidor de descubrimiento de servicios, desde mi API gateway puedo enrutar las peticiones a los diferentes microservicios haciendo uso de los nombres de servicio y no su DNS o IP, lo cual facilita en gran medida el mantenimiento. Además, con esto puedo hacer uso del Balanceador de Carga propio de Spring Cloud Gateway

Así se ve el panel de control de Eureka Server con los servicios registrados

Orquestador

El desarrollo de este orquestador se realizó con el arquetipo de Clean Architecture de Bancolombia, llamado Scaffold Bancolombia. Esto implica el uso del lenguaje de programación Java en su versión 21, junto con el framework más famoso para el desarrollo de aplicaciones en Java, el cual es Spring Framework. Además, con tecnologías como lo son Spring Webflux para el manejo de flujos reactivos y para exponer endpoints y consumir APIs externas funcionando como un Web Client

La ventaja de Scaffold Bancolombia es que me permite estructurar de manera limpia el proyecto, segregandolo por capas, siguiendo las recomendaciones de la Arquitectura Limpia

Simulación de back-end

Para la simulación del back-end de la aplicación, opté por el uso de una gran herramienta con una muy favorable capa gratuita, la cual es Airtable. Con esta herramienta online puedo almacenar data como si se tratase de una base de datos. Puedo simular microservicios para cada entidad, y también tengo a disposición una API con la cual puedo consultar, modificar y eliminar data a placer a través de pticiones HTTP. Para el consumo seguro, esta aplicación ofrece el uso de tokens Bearer con diferentes alcances.
